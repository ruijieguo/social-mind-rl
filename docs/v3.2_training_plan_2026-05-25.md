2. **选项数单一性**: 训练数据 14408 条**全部 4 选项**，模型从未见过 6/15 选项 MCQ → format generalization 失败
3. **Knowledge 饱和**: 3523 条仍只到 0.51 → 进一步加数据 ROI 极低；这部分缺口需要靠 base model capacity 而非数据
4. **小 gap 类** (≤6pp): Desire/Emotion/Intention/Non-literal/SocialIQA/EmoBench EA — 数据质量改善能闭合

## 2. v3.1 训练方案回顾 (Stage 14b)

```yaml
init:        Stage 12 HF (qwen3-14B-tom-hf-stage12)
data:        tom_train_stage14_weighted.jsonl (14408)
max_steps:   200, save_steps: 50
batch:       32 × 8 sample = 256 effective
LR:          1e-6, warmup 20
GRPO + DAPO Clip-Higher (0.20, 0.28)
difficulty_mask: (0.15, 0.80)  # 只训"中等难度"
response_length: 256           # ⚠ 不够 Hi-ToM 长 CoT
prompt_length: 1024
reward: TomMcqRewardWorker (l_min=8, l_max=256)
```

**关键约束 / 限制**:
- `response_length=256` 在 Hi-ToM order_3-4 上**完全不够**（DeepSeek 同题需 1500-4000 token CoT）
- `difficulty_mask (0.15, 0.80)` 把 8 sample 中 6+ 个对的题直接 mask 掉 → 高 order Hi-ToM 在 v3.1 上是 8/8 全错，刚好掉进 mask 范围 (pass rate=0)，但 (0.15, 0.80) 会保留这部分；问题是模型连 1 个对都做不到时无法学习
- 数据**全 4 选项**，无 multi-order belief tracking format
- 训练 token budget 不足: 200 step × 256 tokens × 32 batch × 8 sample = ~13M token

## 3. v3.2 改进方案 (Stage 16)

### 3.1 数据改进 — 关键变更

| 新数据组件 | 数量 | 用途 | 生成方法 |
|---|---|---|---|
| **Hi-ToM-style multi-order** | 1200 | 关键!! 闭合 Hi-ToM order_1-4 gap | bigai-ai/ToM-RL repo 的 generator → 自动生成 N×order×scenario，15 选项 |
| **EmoBench EU_emotion-style** | 500 | 6-opt 情绪命名，闭合 +22pp gap | DeepSeek-v4-pro 合成 (subject 视角，6 个情绪选项) |
| **EmoBench EU_cause-style** | 300 | 4-opt 情绪原因 | DeepSeek-v4-pro 合成 |
| **EmoBench EA-style** | 300 | 4-opt 行动选择 | DeepSeek-v4-pro 合成 |
| **SocialIQA-style** | 400 | 3-opt commonsense | DeepSeek-v4-pro 合成（参考 dev 风格但非测试集） |
| **保留 ToMBench backbone** | 14408 | 维持原能力，避免遗忘 | Stage 14 weighted 全量 |
| **总训练量** | ~17100 条 | 多 format MCQ | 3/4/6/15 选项混合 |

**leakage 检查**: 新合成数据全部经过 MinHash 4-gram Jaccard ≤0.6 与 4 个 benchmark eval 集做去重（复用 `data/tom/dedup_report.json` pipeline）。

### 3.2 训练参数改进

| 参数 | v3.1 (Stage 14b) | **v3.2 (Stage 16)** | 理由 |
|---|---|---|---|
| `init` | Stage 12 HF | **Stage 14b ckpt-199 HF** | 续训 v3.1 |
| `response_length` | 256 | **512** | Hi-ToM 长 CoT 必需 |
| `max_steps` | 200 | **300** | 更多 token budget 应对扩大数据集 |
| `save_steps` | 50 | **30** | 更密 ckpt 捕获 peak |
| `eval_steps` | 25 | **25** | 不变 |
| `difficulty_low` | 0.15 | **0.05** | 允许"几乎全错"题进入梯度（高 order Hi-ToM 起步全错，需要 even 1/8 对就能学） |
| `difficulty_high` | 0.80 | **0.85** | 不变 |
| `learning_rate` | 1e-6 | **5e-7** | 续训需要更小 LR 避免破坏 v3.1 能力 |
| `warmup_steps` | 20 | **30** | 配合新数据分布 |
| `rollout_batch_size` | 32 | 32 | 不变 (内存约束) |
| `l_max` (reward) | 256 | **512** | 匹配 response_length |
| validation | tombench_eval_subset500 | **+ hitom_eval 200 + emobench_eu_emotion 100** | 验证迁移 |

### 3.3 reward worker 改进

`TomMcqRewardWorker` 当前对所有 task 用同一 reward shape (l_min=8, l_max=256, k=50)。Hi-ToM order_3-4 题需要更长 CoT，强行 l_max=256 会惩罚正确长 reasoning → v3.2 改成 task-aware：

```python
# In TomMcqRewardWorker.compute_reward()
if 'hitom' in source.lower() or 'order_' in question_id:
    l_max_local = 512  # allow long reasoning for high-order belief tracking
else:
    l_max_local = self.l_max  # 256 default
```

### 3.4 启动计划

**Phase A (Local Dev, 2-3h)**: 数据合成
- A1: clone bigai-ai/ToM-RL, 跑 Hi-ToM generator 产 1200 records (本地 CPU)
- A2: DeepSeek-v4-pro 合成 EmoBench EU_emotion 500 (并行 conc=8, ~1h)
- A3: DeepSeek-v4-pro 合成 EmoBench EU_cause/EA 600 + SocialIQA 400 (并行)
- A4: MinHash dedup vs 4 个 eval 集 → 输出 `tom_train_stage16.jsonl`

**Phase B (TRAIN Host, 5-7h)**:
- B1: rsync 新数据 + 配置 → TRAIN host (`make sync-up`)
- B2: 启动 stage16 训练，每 30 step 评测 subset500 + hitom200 + emobench100
- B3: 监控 train_monitor，触发停训规则: 任一 val 退化 ≥2pp 即停

**Phase C (Eval, 2h)**:
- C1: 取最优 ckpt 转 HF
- C2: rsync → dev，4 个 benchmark × 3 protocol full eval
- C3: 与 v3.1 / DeepSeek 对比，更新 full_eval_report

**总耗时**: 9-12h，关键路径在 Phase B 训练 (无法并行)。

## 5. 风险与备选方案

### 风险 1: Hi-ToM 数据生成失败
- 风险: bigai-ai/ToM-RL generator 不可用 / 输出格式不匹配
- 应对: Plan B — 用 DeepSeek-v4-pro 直接合成 5-阶 belief 追踪场景（参考 Hi-ToM order_0-4 的 prompt template），手写 generator

### 风险 2: 续训退化 (类似 Stage 13)
- 风险: v3.1 已收敛，在原 ToMBench 数据上续训反而退化
- 应对: 新数据占比 ~16% 强制让梯度流向新分布；监控 train_monitor，step 30 val 退化即停

### 风险 3: response_length=512 显存爆炸
- 风险: max_model_len 翻倍 + vllm GPU mem
- 应对: 降 `actor_infer.gpu_memory_utilization` 0.55 → 0.45；如仍 OOM 改 prompt_length 1024→768

### 风险 4: SocialIQA gap 太小，新数据反而引入噪声
- 风险: SocialIQA gap 仅 +2.25pp，加 400 条合成数据可能过拟合到合成风格
- 应对: SocialIQA 数据放最小比例 (400/17100 = 2.3%)，主要靠 base 能力
